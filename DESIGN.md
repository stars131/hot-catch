# STARTRACE / 星迹前端设计规范

> 状态：锁定。适用于所有产品页面、组件、空态、加载态、失败态与响应式改动。
>
> 前端开发开始前必须完整阅读本文件。现有实现与本文件冲突时，先修订本文件并说明理由，再修改代码。

## 1. 设计立场

星迹采用“编辑部工作台”视觉方向：像一张长期使用的内容工作桌，信息有出处，动作有顺序，状态可追踪。界面以创作任务为中心，装饰保持克制。

三个关键词：

- **清晰**：两秒内识别当前页面、主任务、下一步动作。
- **可信**：任务、作品、发布和失败状态使用准确文案，不用视觉效果掩盖能力边界。
- **有编辑感**：通过版式、细线、索引、留白和排版建立品牌识别，减少卡片堆叠。

设计方法参考 Hallmark 的核心原则：先确定结构，再处理表面；每次交付做反模板化审计；多页面产品共用一套设计系统；移动端和交互状态属于完成条件。

## 2. 产品结构

### 2.1 页面家族

- 创作页：`Workbench`。会话是主界面，Composer 持续可用，Artifact 按需打开。
- 数据与配置页：`Index + Workbench`。顶部标题说明上下文，主体使用列表、表格、分区和必要的表单面板。
- 登录与说明页：`Long document`。控制阅读宽度，用排版和规则线组织内容。

同一产品中的页面共享色彩、字体、按钮语气、圆角、状态和导航。页面差异来自信息结构与内容密度。

### 2.2 创作页固定规则

- 桌面使用一条 `248px` 专用侧栏，组合品牌、主导航、新建创作、会话历史和辅助入口。
- 默认主区保留对话、少量快捷入口和 Composer。
- Artifact 默认关闭；`≥1180px` 时可作为右栏，窄屏全屏覆盖主区。
- 手机创作态不显示全局底部导航，避免遮挡 Composer。
- 快捷入口使用开放式行动索引，禁止回退到四张等宽功能卡。

## 3. 颜色

设计令牌定义在 `app/globals.css`，组件只使用语义令牌。

| 角色 | 令牌 | 当前值 | 用途 |
| --- | --- | --- | --- |
| 页面纸张 | `--color-paper` | `oklch(0.957 0.011 82)` | 页面主背景 |
| 抬升纸张 | `--color-paper-raised` | `oklch(0.982 0.008 82)` | Composer、Popover、Dialog |
| 导航纸张 | `--color-rail` | `oklch(0.925 0.012 78)` | 桌面侧栏、移动抽屉 |
| 主墨色 | `--color-ink` | `oklch(0.205 0.012 72)` | 标题、正文、主图标 |
| 次墨色 | `--color-ink-muted` | `oklch(0.47 0.014 72)` | 说明、时间、元数据 |
| 规则线 | `--color-rule` | `oklch(0.82 0.014 76)` | 分隔线、默认边框 |
| 品牌朱红 | `--color-accent` | `oklch(0.55 0.165 30)` | 主动作、当前状态、关键风险 |
| 焦点苔绿 | `--color-focus` | `oklch(0.48 0.055 148)` | 键盘焦点与少量成功状态 |

约束：

- 单屏品牌朱红面积不超过约 `5%`，常态目标为 `3%`。
- 暖色中性色保持同一色相倾向。
- 文本与背景满足 WCAG：正文至少 `4.5:1`，大字、图标和焦点环至少 `3:1`。
- 品牌朱红填充承载文字时必须使用 `--color-accent-ink`。
- 禁止紫蓝 AI 渐变、渐变文字、霓虹发光、纯黑底、无理由的深色区块。
- 新颜色先新增语义令牌，再进入组件。禁止在页面组件里新增十六进制颜色。

## 4. 排版

- 主体与界面：`Noto Sans SC`，正文 `400`，标签 `500`，标题 `600–700`。
- 元数据：`IBM Plex Mono`，只用于时间、版本、编号、百分比和技术标识。
- 全站最多两套字体家族；Mono 只承担元数据角色。
- 页面标题为 `24–28px`；创作空态标题桌面上限 `32px`。
- 正文最小 `14px`，对话正文 `15px / 1.7`，通用阅读正文建议 `16px / 1.6`。
- 正文行宽控制在 `45–75ch`；对话消息最大宽度 `48rem`。
- 大标题使用负字距 `-0.02em` 至 `-0.04em`；小标签使用 `0.12em–0.18em` 正字距。
- 标题保持直体。强调通过字重、墨色、朱红或细线完成。
- 中文界面使用全角标点；可点击文案保持单行，必要时先缩短文案。

## 5. 空间与容器

- 基础间距为 `4px`，只使用 `--space-*` 或 Tailwind 对应刻度。
- 页面最大宽度为 `1600px`；常规内容区域建议 `1200–1440px`。
- 通过开放式区域、细线、列表和表格组织信息。卡片只用于确实需要独立边界或抬升层级的内容。
- 禁止卡片嵌套、三张等宽功能卡、每个区块都加边框、所有元素使用相同圆角。
- 普通控件圆角 `6–10px`；卡片 `10px`；Composer `18–20px`；圆形只用于明确的圆形控件。
- 阴影只用于 Composer、Popover、Dialog、Sheet 和浮动 Artifact。页面常规卡片使用边框或背景层级。
- 页面分区优先使用规则线、留白节奏和同色系轻微表面变化。

## 6. 组件语气

### 6.1 导航

- 当前项使用轻表面变化、主墨色和左侧 `2px` 朱红标记。
- 图标统一使用项目配置的 Lucide 线性图标，保持相同尺寸与笔画感。
- 主导航行高至少 `40px`，手机触控目标至少 `44px`。
- 品牌区使用星迹轨迹标识、中文名和紧凑英文标识。

### 6.2 按钮

- 主按钮使用朱红实底；次按钮使用纸张底与规则线；Ghost 只服务低层级动作。
- 文案使用明确动词，如“保存更改”“打开作品”“准备发布”。
- 按钮文字保持单行，图标通过 `data-icon` 标注位置。
- `:active` 仅使用 `translateY(1px)`；不使用统一放大、发光或多重悬停效果。

### 6.3 输入与 Composer

- 通用输入高度 `44px`，默认、悬停、焦点和错误状态的边框宽度均为 `1px`。
- 焦点使用 `2px` outline/ring，立即出现，不能参与淡入动画。
- 禁用态同时使用原生 `disabled`、`cursor: not-allowed` 和降低对比度。
- Composer 默认 1–2 行，最多 8 行后内部滚动；平台和发送动作持续可见。
- Composer 是主要抬升面，可使用 `--shadow-float`。外围禁止再套一层卡片。

### 6.4 卡片、列表与状态

- Agent 长回复使用无边框正文；用户消息使用浅暖灰小气泡。
- 进度、参考、成果和审批可使用卡片，因为边界表达了独立状态。
- 空态包含：当前为空、价值说明、一个主要下一步。快捷入口可作为同层索引。
- 加载使用贴合最终结构的 Skeleton；失败在页面内说明原因与恢复动作。
- 成功结果已经在页面可见时保持静默，避免重复 Toast。

## 7. 交互状态

所有交互组件至少实现以下状态：

1. default
2. hover
3. focus-visible
4. active
5. disabled
6. loading（适用时）
7. error（适用时）
8. success（适用时）

状态变化不能改变边框宽度或控件几何尺寸。错误信息保留稳定布局空间，避免出现时推挤页面。

## 8. 动效

- 动效只使用 `transform` 和 `opacity`；颜色与边框可做短时状态过渡。
- 微交互 `120ms`，菜单与小面板 `220ms`，Dialog/Sheet `420ms`。
- 使用 `--ease-out`、`--ease-in`、`--ease-in-out`，禁止 `transition-all`。
- 同一元素每次悬停最多一个主要动效。
- 无限循环只用于加载器等功能状态。
- 每个空间位移动效都要提供 `prefers-reduced-motion` 降级。

## 9. 响应式与无障碍

- 必测宽度：`320 / 375 / 390 / 414 / 768 / 1280 / 1440px`。
- `html` 和 `body` 固定使用 `overflow-x: clip`；禁止 `100vw`。
- 全屏高度使用 `dvh`，并处理 `safe-area-inset-bottom`。
- 图片轨道使用 `minmax(0, 1fr)`；所有弹性子项检查 `min-width: 0`。
- 手机上主要按钮、主导航和 Composer 控件不得换成两行。
- 键盘焦点清晰可见；Dialog、Sheet 保持标题与焦点管理；创作页提供跳转到主内容链接。
- 图标按钮必须提供可理解的 `aria-label`；装饰 SVG 使用 `aria-hidden="true"`。
- 中文、英文和更长语言文案都要检查截断与扩容。

## 10. 反模板化检查

每次前端交付前确认：

- 页面结构与任务一致，未套用营销页 Hero → 三卡 → CTA 模板。
- 没有等宽图标功能卡墙、卡片嵌套、重复圆角容器。
- 没有渐变文字、紫蓝渐变、随机光斑、玻璃拟态、装饰性指标。
- 没有虚构指标、客户标志、评价或成功状态。
- 没有 `transition-all`、批量 `hover:scale-*`、布局属性动画。
- 主操作、页面标题和下一步在第一屏清晰可见。
- Accent 面积、字体数量、正文行宽和对比度符合本文件。

## 11. 开发流程（强制）

任何前端修改都按以下顺序执行：

1. 完整阅读 `DESIGN.md`、`CLAUDE.md`、`CONTEXT.md` 和涉及页面源码。
2. 写清目标流程与要保留的产品能力。
3. 优先复用 `components/ui` 和现有语义令牌。
4. 修改公共组件前检查其所有调用面，避免局部补丁造成全局回归。
5. 运行：

   ```bash
   npm run lint -- --max-warnings=0
   npm run typecheck
   npm run test:unit
   npm run test:contract
   npm run test:integration
   ```

6. 使用真实浏览器检查 `1440×900` 与 `390×844`，至少验证页面身份、空白页、框架错误层、控制台、焦点、一个核心交互和截图。
7. 对照本文件完成反模板化、对比度、状态和移动端检查。
8. 新需求确实需要突破规范时，先更新本文件的“每页例外”，严禁页面内临时覆盖。

## 12. 当前设计基线

- 创作桌面：单一专用侧栏 + 开放式对话画布 + 行动索引 + 底部 Composer。
- 创作移动：紧凑顶栏 + 单栏对话/空态 + Composer；历史与导航进入 Sheet。
- 通用页面：左侧导航 + 顶部标题区 + 内容工作区。
- 设计稿仅用于提取视觉系统，产品 UI 始终由可访问的 React 组件实现。
