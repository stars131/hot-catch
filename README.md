# STARTRACE / 星迹

> 从公开热点与资料检索，到多平台创作、编辑、发布准备和数据复盘的一体化 AI 内容工作台。

[在线体验（邀请制）](https://creat-mcn.wlwl-tools.com) · [部署文档](docs/OPERATIONS.md) · [X 检索说明](docs/x-discovery-and-beta-invites.md) · [平台能力边界](docs/platform-expansion-baseline.md)

## 项目介绍

STARTRACE（星迹）是一个面向个人创作者、内容团队和研究型运营者的开源内容工作台。

它尝试解决的不是“再生成一段文案”，而是内容生产中更长、也更容易失控的一条链路：去哪里寻找正在发生的事情，如何保存来源与证据，怎样把一个选题改写成不同平台真正可用的内容，以及发布后如何继续跟踪和复盘。

在星迹中，热点研究、选题沉淀、多平台创作、版本编辑、发布准备和数据复盘被组织在同一条可追溯工作流里。AI 可以参与检索、归纳、结构设计、改写和评分，但不会隐藏数据来源，也不会把“任务已创建”伪装成“作品已完成”。

## 为什么做星迹

常见的内容工具通常只覆盖其中一个环节：热点榜单、AI 写作、定时发布或数据看板。创作者仍然需要在多个产品、表格和聊天窗口之间搬运资料，并手动判断哪些信息有依据、哪些内容已经保存、哪个版本才是最终稿。

星迹希望提供一条更完整的路径：

```mermaid
flowchart LR
    A[公开热点与资料] --> B[证据化研究]
    B --> C[选题库]
    C --> D[多平台独立创作任务]
    D --> E[版本化编辑中心]
    E --> F[平台预览与发布准备]
    F --> G[发布记录与数据复盘]
    G --> C
```

## 核心能力

### 1. 公开热点与 OSINT 检索

- 聚合 25 个公开热点来源，覆盖新闻、科技、社区和内容平台。
- 支持 X 的地区热点、话题/领域热点和指定博主公开信息收集。
- 默认可使用无需 X 登录态的公开数据入口，也可选配 X 官方只读 API。
- 每条结果保留数据源、采集时间、覆盖范围、原始链接和公开互动指标。
- 单个来源失败不会伪造结果，其余可用来源仍可继续返回。

### 2. 对话式多平台创作

- 通过对话确认受众、内容角度、目标平台、语言和创作 Skill。
- 每个平台建立相互独立的生成任务，可以分别查看、重试和编辑。
- 支持小红书、抖音、YouTube、TikTok、Instagram、X 和 Reddit。
- 支持中文、英语、日语、韩语、西班牙语、法语、德语和巴西葡萄牙语内容。
- 引用资料只作为当前用户的创作上下文，不进入跨用户资料库。

### 3. 平台化编辑中心

- 按平台和具体选题选择内容，不再从聊天记录中寻找最终稿。
- 支持自动保存、撤销/重做、版本历史、历史版本恢复和冲突处理。
- 针对图文、视频脚本、轮播、线程和社区帖子提供不同的结构化编辑器。
- 编辑时同步显示对应平台的内容预览。
- 发布前可保存可见范围、互动权限、内容披露、画面比例、社区规则等平台设置。

### 4. 发布准备与复盘

- 小红书和抖音可通过已配置的发布服务进入真实发布流程。
- YouTube、TikTok、Instagram、X 和 Reddit 当前提供内容包导出与手动发布清单。
- 发布记录使用幂等键，超时后先查询状态，不盲目重复提交。
- 支持公开发布链接跟踪、指标快照和复盘记录。

### 5. 私有部署与限量内测

- 支持 Docker Compose 和 Linux 生产部署。
- 支持带有效期和最大注册人数的共享邀请码。
- 用户凭证使用 AES-256-GCM 加密保存，接口不返回凭证明文。
- PostgreSQL 保存业务状态，Redis 与 BullMQ 执行异步任务。
- 数据库、队列、模型或供应商不可用时明确失败，不回退到伪造的演示数据。

## 平台能力矩阵

| 平台 | 内容形态 | 结构化创作与编辑 | 公开资料导入 | 发布方式 |
| --- | --- | --- | --- | --- |
| 小红书 | 图文 | 支持 | 需配置供应商或授权入口 | 发布服务 |
| 抖音 | 短视频脚本 | 支持 | 需配置供应商或授权入口 | 发布服务 |
| YouTube | 视频内容包 | 支持 | 安全抓取或 Firecrawl | 导出后手动发布 |
| TikTok | 短视频脚本 | 支持 | 安全抓取或 Firecrawl | 导出后手动发布 |
| Instagram | 轮播图文 | 支持 | 安全抓取或 Firecrawl | 导出后手动发布 |
| X | 线程 | 支持 | 公开检索、安全抓取或官方 API | 导出后手动发布 |

“支持创作”不等于“支持自动发布”。尚未完成真实授权、沙盒和生产验收的平台，会明确显示为仅导出，不会生成虚假的发布成功记录。

## 快速开始

### 环境要求

- Node.js 18.18 或更高版本
- Docker Engine / Docker Desktop
- Docker Compose v2

### 全容器启动

```bash
git clone https://github.com/stars131/hot-catch.git
cd hot-catch
cp .env.example .env
```

生成本地凭证加密密钥，并将结果写入 `.env` 的 `CREDENTIAL_ENCRYPTION_KEY`：

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

启动 PostgreSQL、Redis、Web 和 Worker：

```bash
npm run dev:docker
```

打开 [http://localhost:3000](http://localhost:3000)，健康检查地址为 [http://localhost:3000/api/health](http://localhost:3000/api/health)。本地环境默认启用开发账号旁路；生产环境必须关闭旁路并配置正式认证。

更细的本地开发、真实凭证、生产部署、备份和恢复步骤见：

- [内测与本地开发手册](docs/internal-beta.md)
- [生产运维手册](docs/OPERATIONS.md)

## 设计与安全原则

- **证据优先**：公开检索结果保留原始链接和采集边界，便于复核。
- **能力透明**：区分任务入队、内容生成、保存完成和发布完成。
- **人工最终负责**：AI 输出、社区规则、素材权利和发布设置都应在发布前人工检查。
- **不绕过平台访问控制**：不读取私密账号，不使用账号池、Cookie 轮换或代理池规避限制。
- **用户数据隔离**：业务查询必须验证当前用户归属，凭证以密文保存。
- **失败不伪装成功**：上游不可用时返回明确错误或部分结果，不用假数据填充页面。

## 当前状态

项目处于邀请制 Beta 阶段，主要工作流已经可以自托管运行，但公开数据源和第三方平台能力会随上游规则变化。用于生产环境前，请自行评估所在地区的法律法规、目标平台条款、数据授权范围以及内容合规要求。

目前适合：

- 独立创作者和小型内容团队搭建自己的创作工作台；
- 研究公开热点、行业话题和公开博主内容的运营人员；
- 希望研究多 Agent 内容工作流、证据链和版本化编辑的开发者；
- 需要邀请制、私有数据和自托管能力的内部工具团队。
